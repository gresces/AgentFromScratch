# AgentFromScratch 维护规范

这个技能用于维护 `AgentFromScratch` 项目。项目目标是使用 C++ 构建一个高性能、可拓展、个人使用的 Agent：先完成可在终端运行的 Agent 核，再在其上构建展示层。

## 适用场景

在执行以下工作前使用本技能：

- 新增、修改、重构或删除项目代码。
- 调整目录结构、构建系统、依赖、工具链或配置。
- 更新 `AGENTS-CN.md`、`DEV-STATUS.md`、`TODO.md` 等描述性文件。
- 维护开发状态、拆分任务、整理后续计划。
- 准备提交、同步或推送到远程仓库。

## 项目信息

- 项目名称：`AgentFromScratch`
- 主要语言：C++
- 远程仓库：`https://github.com/gresces/AgentFromScratch`
- 首要交付物：可在终端作为二进制程序运行的 Agent 核。
- 后续交付物：基于 Agent 核构建的展示层。

## 最高优先级约束

- 所有代码编写、修改、重构和评审必须遵守同目录下的 `../coding-standards/` 规范。
- 修改 C++ 代码时，必须同时遵守 `../coding-standards/language-rules/cpp/format.md`。
- 项目描述性文件必须使用中文。
- 不为假想需求提前设计复杂抽象；只有当前需求证明必要时才增加扩展点。
- 优先保证 Agent 核的清晰边界、可测试性和稳定性；展示层不得污染核心逻辑。
- 不提交占位实现、空壳接口、假回退、无效错误处理或未验证的行为。

## 文档约定

### `AGENTS-CN.md`

- 作为代码描述文件存在于项目的多个目录层级。
- 下级目录的 `AGENTS-CN.md` 继承上级目录约定，只补充本目录新增的职责、边界、依赖和注意事项。
- 不重复顶层已经明确的通用规则。
- 应记录稳定约定，不记录容易过期的临时进展。

### `DEV-STATUS.md`

- 仅在必要目录出现。
- 用于记录当前目录的开发状态、已完成内容、阻塞点、近期决策和风险。
- 内容必须简短、事实化、可维护。
- 当状态过期时，应同步更新或删除过期段落。

### `TODO.md`

- 仅在项目顶层出现。
- 用于记录后续任务。
- 任务结构固定为两级：任务和子任务。
- 不把已经完成或已经废弃的任务长期保留为待办。

## 维护流程

### 1. 理解范围

- 先阅读当前目录及父级目录的 `AGENTS-CN.md`。
- 如果目录存在 `DEV-STATUS.md`，先确认当前状态和阻塞点。
- 如果工作影响后续计划，检查顶层 `TODO.md`。
- 修改代码前，确认相关模块的职责边界和现有约定。

### 2. 设计变更

- 优先做最小正确变更。
- 保持 Agent 核与展示层解耦。
- 保持输入输出、工具调用、上下文管理、状态持久化等边界明确。
- 新增公共接口前，先确认是否已有等价能力。
- 删除代码时确认没有仍在使用的调用点。

### 3. 编写代码

- 遵守 `../coding-standards/` 的通用规范。
- C++ 代码应优先使用明确所有权、清晰生命周期和低额外分配的设计。
- 错误信息必须能定位失败位置和原因，不泄露密钥、令牌或私人数据。
- 注释解释原因、边界和不变量，不复述代码字面行为。
- 不引入无必要依赖；标准库和现有依赖优先。

### 4. 更新文档

- 行为、目录职责或维护约定改变时，同步更新相应 `AGENTS-CN.md`。
- 当前目录状态发生实质变化时，同步更新 `DEV-STATUS.md`。
- 新增后续工作时，只在顶层 `TODO.md` 记录，并保持两级任务结构。
- 文档只写已确认事实；推测内容必须明确标注为待确认。

### 5. 验证变更

- 优先运行与变更直接相关的构建、格式化、静态检查和测试。
- C++ 代码变更至少应通过可用的格式化检查、编译检查或最小可运行场景。
- 文档变更至少应重新读取目标文件，确认内容完整、中文表达一致、层级规则正确。
- 如果无法运行验证，必须记录具体原因和已完成的替代检查。

### 6. 远程协作

- 默认远程仓库为 `https://github.com/gresces/AgentFromScratch`。
- 推送前确认本地变更范围清晰、验证结果可复述、描述性文件已同步。
- 不在提交中混入无关格式化、临时调试输出或未说明的大规模移动。
- 不提交密钥、令牌、本地路径、机器特定配置或私人数据。

## 编译、运行和测试流程

- 当前 Xmake 项目入口是 `core/xmake.lua`，构建、运行和测试命令默认在 `core/` 目录执行。
- 使用 C++23 标准（`set_languages("c++23")`），所有代码应兼容 C++23 编译器。
- 编译主程序使用 `xmake`，首次需加 `-y` 自动安装依赖。编译产物统一输出到仓库顶层 `bin/` 目录。
- 运行主程序使用 `xmake run Agent`。运行时工作目录由 `core/xmake.lua` 的全局 `set_rundir` 指向仓库顶层 `bin/` 目录。
- 运行测试使用 `xmake test`。如果后续新增测试目标，测试目标产物也应输出到仓库顶层 `bin/` 目录；未单独配置 `rundir` 的测试应继承目标的 `set_rundir`，测试时工作目录保持在 `bin/`。
- 检查目标输出路径使用 `xmake show -t Agent`，确认 `targetfile` 指向仓库顶层 `bin/` 下的可执行文件。
- 当前没有测试目标时，`xmake test` 输出 `nothing to test` 属于预期结果；新增测试后必须运行能覆盖变更的具体测试。
- 每次构建自动生成 `core/compile_commands.json`，供 LSP（clangd 等）进行静态分析。

## 项目依赖

依赖通过 Xmake 内置的包管理器 `xrepo` 管理，在 `core/xmake.lua` 中声明：

- `add_requires` 在全局作用域声明包名。
- `add_packages` 在 `target` 作用域内将包链接到目标。

当前依赖：

| 包名 | Xmake 包名 | 版本 | 用途 |
|------|-----------|------|------|
| [nlohmann/json](https://github.com/nlohmann/json) | `nlohmann_json` | 最新 | JSON 解析与序列化 |
| [FTXUI](https://github.com/ArthurSonzogni/FTXUI) | `ftxui` | 最新 | 终端用户界面 |
| [Boost.SML](https://github.com/boost-ext/sml) | `boost_sml` | 最新 | Agent 核心状态机 |
| [Taskflow](https://github.com/taskflow/taskflow) | `taskflow` | 最新 | 状态内部任务并行执行 |
| [cpr](https://github.com/libcpr/cpr) | `cpr` | 最新 | HTTP 客户端（模型 API 调用） |

新增依赖时，先通过 `xrepo search <keyword>` 确认包名，在 `core/xmake.lua` 的 `add_requires` 和 `add_packages` 中分别添加，再运行 `xmake -y` 安装并编译验证。

## 维护者检查清单

完成一次维护工作前确认：

- 已读取相关 `AGENTS-CN.md`，并遵守层级约定。
- 代码变更符合 `../coding-standards/` 和对应语言规范。
- Agent 核边界没有被展示层、临时脚本或外部接口污染。
- 新增行为有对应验证，验证结果可以复述。
- 描述性文件仍为中文，且没有过期状态或错误待办。
- 远程仓库地址、提交范围和敏感信息检查无误。
